Stryker S3

Error: Function Disabled

Asset Type

Hospital Bed

Manufacturer

Stryker

Model

S3

What This Guide Helps With

Troubleshooting Function Disabled - Call Service when Scale, Bed Exit, or iBED Awareness operation is blocked by an active logged fault. This condition means the affected function cannot be trusted for clinical monitoring or safety-related status information until the underlying fault is identified and corrected.

Step-by-Step Troubleshooting

1. Ensure Patient Safety First

  • Do not rely on the affected Scale, Bed Exit, or iBED Awareness function while this message is active.
  • If the bed is occupied and the disabled function is required for patient safety, coordinate alternate monitoring or fall-prevention measures before beginning troubleshooting.
  • Move the patient to an appropriate backup bed or alternate monitoring arrangement before evaluating a function required for active patient care.

Expected outcome: The patient is protected by appropriate alternative measures while the disabled bed function is evaluated.

Why it matters: A disabled Scale, Bed Exit, or iBED Awareness function cannot be trusted to provide accurate monitoring or status information.

2. Verify the Reported Error and Behavior

  • With the bed removed from patient-dependent use as appropriate, attempt to reproduce the reported condition.
  • Observe whether Function Disabled - Call Service blinks when attempting to activate Scale, Bed Exit, or iBED Awareness.
  • Document which function request causes the message to appear and whether more than one function is affected.

Expected outcome: The message is reproduced when attempting to use a specific disabled function.

Why it matters: This confirms that operation is being blocked by a fault affecting the requested system rather than a simple user-setting issue.

3. Check Basic External Conditions

  • Confirm the bed has stable AC power and no obvious power interruption or damaged power connection.
  • Inspect accessible cables, connectors, siderails, footboard connections, and control interfaces for damage, looseness, pinching, contamination, or fluid intrusion.
  • Check that accessories, linens, cords, or nearby equipment are not interfering with the weighing structure, siderails, or bed frame.
  • Confirm the bed is on a stable surface and is not contacting walls, equipment, or obstacles that could affect scale-related operation.
  • Manually verify siderail positions and locks if iBED Awareness status is affected.

Expected outcome: External interference, visible damage, or disconnected components are identified and corrected or ruled out.

Why it matters: An underlying load-cell, communication, zeroing, or siderail-switch fault may sometimes be related to an observable external condition.

4. Identify Which Function Is Disabled

  • If Scale cannot be activated, suspect an active scale-system fault.
  • If Bed Exit cannot be activated, suspect an active Bed Exit or related scale-system fault.
  • If iBED Awareness cannot be activated, suspect an active Awareness-system fault, such as a siderail status issue.
  • Do not assume the disabled function itself is the failed component; it may be blocked because another logged condition affects safe operation.

Expected outcome: The affected system is identified so further troubleshooting can be directed appropriately.

5. Review the Error Log

  • Use the appropriate Maintenance Menu function to review the bed Error Log for the underlying logged error associated with the disabled function.
  • This displayed message may be associated with 901 ScaleSystem, 902 BedExitSystem, or 903 LBSSystem.
  • Look for a more specific logged condition, such as a load-cell out-of-range fault, scale zeroing failure, scale communication fault, or siderail switch fault affecting iBED Awareness.

Expected outcome: A specific underlying error is identified that explains why the function has been disabled.

Why it matters: Function Disabled - Call Service is a system-level message. Correct troubleshooting depends on identifying and addressing the actual logged fault behind it.

6. Isolate External or Accessory-Related Causes

  • For a scale or load-cell-related fault, check for frame interference, unstable positioning, damaged accessible wiring, or objects affecting the weighing structure.
  • For a communication-related fault, inspect accessible harness connections and wiring for looseness or visible damage.
  • For an iBED Awareness or siderail-related fault, inspect the affected siderail position, locking action, switch actuation area, and accessible wiring.
  • Correct only confirmed external conditions and then reassess the bed.

Expected outcome: A visible or accessible contributing cause is corrected, or the problem is isolated as likely device-related.

7. Verify the Corrected Function

  • After the identified source fault has been corrected according to service procedure, test the previously disabled function.
  • Verify that Scale activates and provides stable operation, if affected.
  • Verify that Bed Exit can be armed and responds appropriately during functional testing, if affected.
  • Verify that iBED Awareness accurately recognizes monitored bed conditions, including siderail status where applicable, if affected.
  • Confirm that Function Disabled - Call Service no longer appears when the function is activated.

Expected outcome: The affected function operates normally without the message returning.

Why it matters: Clearing the message alone does not confirm repair. The affected patient-safety-related function must operate correctly before return to service.

8. Stop When Resolved

  • If an external condition or corrected underlying logged fault resolves the message and the affected function passes operational verification, document the repair and return the bed to service according to department procedure.

If the Problem Persists

If Function Disabled - Call Service remains active after basic external checks and identification of the underlying logged fault, external causes have been ruled out and the disabled Scale, Bed Exit, or iBED Awareness function should be considered unreliable.

Persistent system-disabled messages may indicate an unresolved device fault, such as a failed load cell, communication circuit issue, or failed sensing component.

Remove the bed from service, label it Out of Service, and send it for repair or bench evaluation.

Knowing when to stop is proper troubleshooting.

Clinical Use Tip

Never troubleshoot on an active patient. Move the patient to a backup device or provide approved alternate monitoring and fall-prevention measures first. Never rely on Bed Exit, Scale, or iBED Awareness monitoring while the bed reports that the function is disabled.

Work Order Documentation (CCR Method)

CCR = Complaint, Cause, Resolution

Complaint

What was reported by the clinical staff.

Example:
"Stryker S3 displayed Function Disabled - Call Service when attempting to activate Bed Exit. Bed Exit function could not be enabled for clinical use."

Cause

What was observed during troubleshooting.

Example:
"Reviewed Error Log and identified an active logged system fault affecting the Bed Exit function. External inspection completed; no accessory interference or visible connection issue corrected the condition. Underlying device fault remains suspected."

Resolution

What action was taken.

Example:
"Confirmed Bed Exit could not be relied upon, notified clinical staff to use alternate patient-safety measures, removed bed from service, labeled unit Out of Service, and referred for further repair evaluation."

Helpful Details to Include (If Known)

  • Alarm behavior
  • Accessories swapped
  • Power behavior
  • Environmental factors
  • Indicator lights
  • Final device status

Final Thought

A Function Disabled - Call Service message means the bed has intentionally blocked a safety-related function because an underlying fault is present. Patient safety depends on recognizing that the disabled function cannot be trusted, logically identifying the source error, correcting only confirmed issues, and escalating unresolved conditions with clear documentation.

That is successful troubleshooting.
